# Changelog

Living record of what's been built, grouped by the spec's build phases. Updated at the end of each phase тАФ entries are amended in place if a later phase changes something an earlier entry described, not just appended to. If you're picking up this project cold, read this top to bottom before touching code.

## Phase 1 тАФ Foundation

Laravel 13.19 + Livewire 4.3 + Filament 5.6 + spatie/laravel-permission 8.3, scaffolded and verified. Filament admin panel at `/admin` with a light theme and teal accent per the spec's UI requirements. MySQL database, admin user seeded.

Deviations from the original spec worth knowing:
- Spec assumed Laravel 11 / Livewire 3 тАФ those are past security support as of mid-2026, so latest-stable was used throughout instead (Laravel 13, Livewire 4, Filament 5).

## Phase 2 тАФ Core CRM: Agents, Companies/Contacts, Leads, Pipeline, Activities/Follow-ups

- **Roles & agents**: `User` model gained spatie roles (`admin`/`agent`/`production`), plus `work_scope` (`lead_gen_only`/`sales_only`/`full_cycle`), `status` (active/inactive), `created_by_admin_id`. New **Agents** resource lets Admin create/edit/deactivate agents and assign work-scope.
- **Companies & Contacts**: full CRUD, one company has many contacts, company profile page shows contacts + lead history.
- **Leads**: full lifecycle model with `LeadActivity` timeline (calls, notes, emails, status changes, reassignments), follow-up date/note fields.
- **Lead Kanban pipeline board** (NewтЖТContactedтЖТQualifiedтЖТQuotedтЖТNegotiationтЖТWonтЖТLost), built on `relaticle/flowforge` (the only Kanban plugin that installs cleanly on Laravel 13/Filament 5 тАФ others pin `illuminate/contracts` to v11/v12). This is the board from spec ┬з3.1, distinct from the separate Deal Board added in Phase 3.
- **Role-scoped visibility**: Admin sees every lead; Agents see only their own assigned leads; `sales_only` agents can't create new leads; Production has zero access to the sales pipeline. Enforced via `LeadPolicy` + scoped `getEloquentQuery()`.
- **Business rules**: `lead_gen_only` agents are blocked from dragging a lead past "Qualified" (spec's hand-off rule); moving to "Lost" requires a reason via a dedicated action, not a bare drag; every status change/reassignment logs a `LeadActivity`; Admin can reassign leads to a different agent.
- **Follow-ups dashboard widget**, scoped by role.

Bugs hit and fixed during this phase:
1. Filament 5 moved the `Get` form-closure type-hint from `Filament\Forms\Get` to `Filament\Schemas\Components\Utilities\Get` тАФ the old namespace throws a `TypeError` only visible in `storage/logs/laravel.log`, not on screen.
2. `relaticle/flowforge` v4.0.13 ships without its compiled CSS and never registers a stylesheet asset тАФ its board rendered as an unstyled vertical stack. Fixed via Filament's official custom-theme mechanism (`resources/css/filament/admin/theme.css`, registered via `->viteTheme()`), which builds our own Tailwind bundle that scans the plugin's Blade views. This is the general fix for any Filament plugin that renders unstyled.

## Phase 3 тАФ Quotations & Products

- **Product Catalog**: `Product` (name, type, material, size options, MOQ, base price) with `ProductPriceTier` line items (quantity breakpoints тЖТ unit price), managed via a `Repeater` on the product form. Admin manages the catalog; Agents get read-only visibility (spec ┬з3.5).
- **Deal Board** (spec ┬з3.3 тАФ the sales-pipeline board, separate from the Lead board): a `Deal` is auto-created the moment a Lead's status reaches "Quoted" (hooked into the existing `LeadsBoard::moveCard()` and `EditLead` save, additively тАФ the Lead board itself is untouched). Its own Kanban (Quoted тЖТ Negotiation тЖТ Won тЖТ Lost) via the same Flowforge pattern as the Lead board, with value/expected-close-date/probability on each card and filters by agent/product/value range. Scoped the same way as Leads (Admin sees all, Agent sees own, Production has none).
- **Quotations**: built as a relation manager on a Deal's page тАФ line items pick a Product and auto-fill unit price from whichever price tier matches the entered quantity (reactive, updates live as you type). Versioning via a "Duplicate as new version" action. Status flow Draft тЖТ Sent тЖТ Approved/Rejected. A configurable discount-approval threshold (Admin-editable at **System тЖТ Settings**) blocks "Sent" until an Admin explicitly approves the discount if it exceeds the threshold. PDF download and a 14-day signed share link (no login required) via `barryvdh/laravel-dompdf`.
- **Deal-to-Order conversion**: marking a Deal "Won" creates a minimal `Order` row (status `Pending`) and syncs the parent Lead to "Won". The full order/dispatch *workflow* (status pipeline, Production role UI, dispatch form) is Phase 4 тАФ this phase only lays the data down, per the spec's own phase split ("Deal-to-Order conversion" is explicitly listed under Phase 3, the *pipeline* under Phase 4).
- **System Settings** page (Admin-only): currently just the discount-approval threshold; built to hold more settings as later phases need them (e.g. pipeline stage naming, mentioned in spec ┬з3.11, intentionally deferred тАФ see "Known gaps" below).

Bugs hit and fixed during this phase:
1. Same `Get`-namespace issue as Phase 2, but for `Set`: `Filament\Forms\Set` тЖТ `Filament\Schemas\Components\Utilities\Set`. Hit in the quotation line-item repeater's reactive price autofill. Worth grepping for `use Filament\\Forms\\(Get|Set);` whenever a `TypeError` mentions these classes тАФ that's always the fix.
2. **Filament 5 defaults relation managers on a resource's View page to read-only** (`hasReadOnlyRelationManagersOnResourceViewPagesByDefault()`), hiding Create/Edit/Delete and showing only View тАФ silently, with no error. This broke the Quotations relation manager (no "New quotation" button appeared) *and*, it turned out, had been silently affecting Phase 2's Contacts relation manager on a Company's profile page the whole time (adding a contact from a company's page never worked, and nothing exercised that path until now). Fixed both by overriding `isReadOnly(): bool { return false; }` on `QuotationsRelationManager` and `ContactsRelationManager`. **If a relation manager on a `ViewRecord` page is ever missing its Create/Edit buttons with no visible error, this is almost certainly why.**
3. `Deal::firstOrCreate(['lead_id' => ...], [...])` without explicitly setting `stage` in the create array left the in-memory model's `stage` attribute `null` even though the DB row correctly got the column's default value тАФ Eloquent doesn't re-fetch DB-applied defaults after an insert. Harmless where the returned instance wasn't used further, but fixed by passing `'stage' => DealStage::Quoted` explicitly at both call sites so the in-memory object is always trustworthy immediately after creation.

Verification for this phase used real logged-in browser sessions (Playwright + Edge) covering: product creation with a price tier, the Deal board rendering and scoping, building a quotation with a reactive product/quantity/price line item, the discount-approval gate, the PDF route, "Mark as Sent", and the WonтЖТOrder conversion тАФ plus a full smoke test loading every nav page as all four roles (Admin, and agents of each work-scope) with zero HTTP 500s or console errors, and a regression pass confirming the Phase 2 Lead board and its seeded data were untouched.

### Known gaps deliberately left for later phases
- Pipeline stage *naming* is not yet configurable (spec ┬з3.11) тАФ stages are currently fixed PHP enums (`LeadStatus`, `DealStage`, `OrderStatus`). Making labels admin-editable would mean moving them into the database, which is a bigger structural change than this phase's scope; flagged here so it isn't forgotten, not silently dropped.
- A company-wide `activity_log` covering *every* admin action (spec ┬з3.11) doesn't exist yet тАФ only Lead-specific actions (status changes, reassignment) are logged, to `lead_activities`. General accountability logging is Phase 7 (Polish) territory.

**Amended in Phase 6**: `Deal.value` was originally a manually-typed field with no connection to the quotation actually built for that deal тАФ meaning a deal's value would silently stay blank (or wrong) unless someone remembered to copy the quotation total across by hand. This surfaced as a real bug while building Phase 6's revenue reporting (see below): the seeded example deal had a 5,000 quotation but a blank `value`, so every revenue widget read zero. Fixed by syncing `Deal.value` from `Quotation::recalculateTotals()` whenever the quotation being recalculated is the deal's current (latest-version) quotation. The manual `value` field on the Deal form still exists and still works for deals that don't have a quotation yet тАФ the sync only takes over once one exists.

## Phase 4 тАФ Production & Dispatch

- **Order queue**: `OrderResource` (nav group "Production") is where the Order rows created in Phase 3 actually get worked. Admin sees every order; the owning Agent sees their own orders (via `deal.sales_rep_id`) as a status readout; **Production sees every order with no sales-pipeline access at all** (spec ┬з2C) тАФ Production's nav has no Leads/Deals/Companies/Contacts/Products, only Orders, and the order table/infolist never surfaces price/quotation figures to keep with "Production cannot see pricing unless Admin allows it."
- **One-click status pipeline** (Pending тЖТ In Production тЖТ Ready to Dispatch тЖТ Dispatched тЖТ Delivered): each stage is a single button visible only to Production/Admin. Moving to "Dispatched" requires the **dispatch form** first (vehicle/driver info, dispatch date, delivery address, invoice/tracking number тЖТ `Dispatch` model, one-to-one with the order).
- **In-app notifications with a topbar bell icon** (spec ┬з3.9, first concrete instance): every status transition sends a Filament database notification to both the person making the change (a toast) and the order's sales rep (persisted to the bell). Enabled via `->databaseNotifications()` on the panel тАФ this is Filament's built-in mechanism, no custom UI needed.
- **Orders overview widget**: pending/in-production/ready/dispatched counts, visible to Admin and Production on the dashboard (spec ┬з3.10's Production-facing dashboard requirement тАФ pulled forward from Phase 6 since it's tightly coupled to this phase's own data, not cross-cutting business analytics).

Bugs hit and fixed during this phase:
1. **`QUEUE_CONNECTION=database` with no queue worker ever running meant every Filament database notification silently vanished into the `jobs` table instead of being delivered** тАФ `sendToDatabase()` reported success, no exception was thrown anywhere, and the bell icon just never showed anything. This is easy to miss because nothing *fails visibly* тАФ the only tell is `unreadNotifications()->count()` staying at 0 while `DB::table('jobs')->count()` climbs. Fixed by switching local dev to `QUEUE_CONNECTION=sync` (processes jobs immediately, inline тАФ no separate worker process to remember to start, consistent with keeping this dev environment's number of "must remember to run" background processes as low as possible). If a queue *is* wanted later (e.g. for genuinely slow jobs), remember `php artisan queue:work <connection>` needs the connection name explicitly if it's not the default тАФ `queue:work` alone only drains whatever `QUEUE_CONNECTION` currently points to.

Verification: full Production pipeline walked end-to-end via a real logged-in Production-role browser session (Pending тЖТ In Production тЖТ Ready to Dispatch тЖТ dispatch form submission тЖТ Dispatched тЖТ Delivered), confirmed the sales rep actually received each notification (checked `unreadNotifications()` directly, not just the toast), confirmed Production's nav has zero sales-pipeline items, and re-ran the full Phase 2/3 regression + all-roles smoke test тАФ zero errors.

## Phase 5 тАФ Internal Team Chat

- **Real-time messaging** via `laravel/reverb` (WebSocket server) + `laravel-echo`/`pusher-js` on the client. New **Chat** nav item (own top-level page, not a resource) with a two-column layout: Direct Messages (every active user except yourself, green/gray dot for online/offline, unread-count badge) and Group Channels (`# Sales Team`, `# Production Updates` тАФ fixed channels for now, not yet admin-configurable).
- **`messages` table**: `sender_id`, nullable `receiver_id` (DMs) or `channel` string (groups), `content`, `read_at`. A DM and a group message are mutually exclusive on one row (`receiver_id` set тЖТ DM, `channel` set тЖТ group).
- **Online status**: `TrackLastActive` middleware stamps `users.last_active_at` on every authenticated request (throttled to once/minute to avoid a write per request); `User::isOnline()` checks it's within the last 3 minutes. Shown as the dot next to each DM contact.
- **Unread badges**: count of messages from that sender with `read_at IS NULL`; opening a DM marks them read.
- **Admin chat log viewer**: read-only `MessageResource` (nav group "System", labeled "Chat Logs") тАФ Admin can audit any DM or channel message, no create/edit/delete.
- **Fallback delivery**: `wire:poll.10s` on the Chat page means a message is *never* more than 10 seconds late even if the real-time path fails for any reason тАФ real-time is a latency improvement on top of a guaranteed-eventual-consistency baseline, not the only delivery path.

### Environment additions
`php artisan reverb:start` is a **new required long-running background process**, same category as MySQL and `artisan serve` тАФ it does not run itself, and if it's not running, `window.Echo` still connects fine but nothing ever arrives (Reverb is the actual WebSocket server, not the client library). Start it from the repo root:
```
php artisan reverb:start
```
New `.env` keys (real values generated for this dev environment, regenerate for any other machine тАФ see `.env.example` for the placeholders): `BROADCAST_CONNECTION=reverb`, `REVERB_APP_ID`, `REVERB_APP_KEY`, `REVERB_APP_SECRET`, `REVERB_HOST=localhost`, `REVERB_PORT=8080`, `REVERB_SCHEME=http`, plus the `VITE_REVERB_*` mirrors the frontend build reads at compile time (Vite inlines `.env` values at build, so `npm run build` must be re-run after changing any `VITE_REVERB_*` value). `QUEUE_CONNECTION` was already `sync` since Phase 4 тАФ broadcast events (`ShouldBroadcast`) queue through the same connection, so they fire immediately with no separate worker needed, consistent with the Phase 4 notification fix.

### Bugs hit and fixed during this phase
1. **Duplicate `BROADCAST_CONNECTION` in `.env`**: `php artisan install:broadcasting --reverb` appended a second `BROADCAST_CONNECTION=reverb` line at the bottom of `.env` without removing the original `BROADCAST_CONNECTION=log` above it. Laravel's dotenv parser keeps the *first* occurrence of a key, so the old `log` value silently kept winning and no broadcaster ever activated. Fixed by deleting the duplicate. **Whenever an `install:*` Artisan command is used on a `.env` that already defines the same key, check for duplicates afterward** тАФ the command appends, it doesn't merge.
2. **Reverb credentials never generated**: the same install command left `REVERB_APP_ID`/`REVERB_APP_KEY`/`REVERB_APP_SECRET` blank, which surfaced as `Pusher\Pusher::__construct(): Argument #1 ($auth_key) must be of type string, null given` the moment anything touched `routes/channels.php` (even `php artisan list`). Fixed by generating and setting values manually.
3. **The single biggest bug of this phase: the compiled JS bundle containing Echo was never actually loaded into the Filament admin panel**, so `window.Echo` was `undefined` on every page and real-time delivery silently never worked тАФ not a connection failure, not a channel-auth failure, the script tag simply wasn't on the page. `AdminPanelProvider`'s `->viteTheme(...)` only wires up the CSS theme; it has nothing to do with loading `resources/js/app.js` (the entry point that does `import './echo'`). **Any Filament panel that needs its own JS (Echo, or any other frontend library) needs an explicit render hook** тАФ fixed by adding:
   ```php
   ->renderHook(
       PanelsRenderHook::SCRIPTS_AFTER,
       fn (): string => Vite::withEntryPoints(['resources/js/app.js'])->toHtml(),
   )
   ```
   in `AdminPanelProvider::panel()`. This one was caught by literally checking `window.Echo` from a real browser session тАФ reading the code gave no reason to suspect this, since `resources/js/echo.js` and its import chain all looked correct in isolation.
4. **Even after fixing #3, real-time delivery still silently didn't work** тАФ messages were correctly saved and correctly broadcast (confirmed by subscribing to the raw private channel with plain `Echo.private(...).listen(...)`, bypassing Livewire entirely, which *did* receive the event), but the Chat page's own `getListeners()`-based dynamic echo binding never fired. Root cause, found by reading Livewire's own source (`vendor/livewire/livewire/src/Features/SupportEvents/SupportEvents.php`): the `listeners` effect тАФ the payload that tells the client-side Livewire runtime which Echo channels to bind тАФ is **only ever sent to the browser during the component's initial mount response**, never on subsequent partial-update responses (`if ($context->isMounting() || ...)`). Since `Chat::getListeners()` returns `[]` at mount (no conversation is selected yet on first page load), and only returns the real channel name *after* the user clicks a DM/group (a subsequent, non-mount request), the updated listener list was computed correctly on the server but never reached the client at all. **This means Livewire's classic `protected function getListeners()` mechanism cannot support a channel name that becomes known only after mount тАФ it is mount-time-only, full stop**, regardless of how correctly the dynamic channel name itself is computed. Fixed by abandoning that mechanism entirely and driving the Echo subscription from JavaScript instead: a public `$activeChannel` property is set in `selectDm()`/`selectGroup()`, and a `@script` block in `chat.blade.php` calls `$wire.$watch('activeChannel', ...)` to subscribe/unsubscribe from the correct private channel reactively, forwarding received events into the component via `$wire.onMessageReceived(event)`. **If a future page needs an Echo channel whose name depends on component state set after mount, this `$wire.$watch` + `@script` pattern is the correct approach тАФ `getListeners()` is not.**
5. Not a bug, but a real characteristic worth documenting: there's a **small window (typically under ~1-2 seconds) between a user opening a conversation and their private-channel WebSocket subscription being confirmed by Reverb**. A message sent by the other party inside that window can be missed by the real-time path тАФ this was caught by a Playwright test that clicked a channel and sent a message from the other side almost immediately, which failed intermittently, while the same test with a `waitForFunction` on `channel.subscribed === true` before sending passed every time. This is inherent to any pub/sub WebSocket system, not something to "fix" тАФ it's exactly why `wire:poll.10s` exists as a fallback (see above), so the absolute worst case is a 10-second delay, never a lost message.

Verification for this phase used real two-browser-session Playwright tests (not just single-session reload checks): confirmed messages persist correctly (checked directly via `php artisan tinker`, independent of the UI), confirmed a DM reply round-trip renders on both sides, confirmed genuine real-time delivery with no page reload for both a DM (`admin тЖТ fiona`, delivered in ~2.5s) and a group channel (`admin тЖТ sales-team`, delivered in ~2.2s, after confirming subscription first), confirmed `Echo.private(...).listen(...)` receives the raw broadcast independent of Livewire (isolating broadcasting infra from the page's own wiring), confirmed the admin-only Chat Logs resource and its 403-for-agents access control, and re-ran the full all-roles smoke test (now covering `/admin/chat` for every role and `/admin/messages` for Admin only) plus the Phase 2-4 regression тАФ zero HTTP 500s, zero console errors, zero `storage/logs/laravel.log` entries during the entire verification run (the two `pageerror: Object` and two 403-console-log lines the smoke test reports are expected: the former is the test browser closing while a `wire:poll` request from a *previous* test script was still in flight тАФ confirmed harmless by re-running with a settle delay before closing, which produced zero errors тАФ and the latter is the test *deliberately* checking that a 403 is what an agent gets, not a 500).

## Phase 6 тАФ Reporting

- **Admin dashboard**: `RevenueStatsWidget` (total revenue from Won deals, revenue this month, win rate, average order turnaround in days from Won to Delivered), `RevenueByAgentChart` and `RevenueByProductChart` (bar charts), `RevenueByMonthChart` (line chart, trailing 6 months), `LostReasonChart` (doughnut breakdown of why deals were lost), `TopClientsWidget` (top 10 companies by Won-deal revenue), `AgentConversionWidget` (won/lost/conversion % per agent). All seven are Admin-only (`canView()` gated the same way `OrdersOverviewWidget` already was in Phase 4) and appear automatically on the existing Dashboard page via Filament's panel-wide widget discovery тАФ no page changes needed.
- **Agent dashboard**: `MyPipelineWidget` (own open pipeline value тАФ Quoted + Negotiation deals тАФ and own conversion rate). Visible to any `agent`-role user; the Phase 2 Follow-ups widget is untouched and still shows alongside it.
- **Production dashboard**: unchanged тАФ the Phase 4 `OrdersOverviewWidget` already covers this role's needs, per that phase's own note about pulling the widget forward.
- **New `orders.delivered_at`** column, stamped by the existing "Mark delivered" action in `OrdersTable`, so turnaround can be measured precisely (Won/Order-created тЖТ actually Delivered) instead of relying on `updated_at`, which would have been wrong the moment anyone edited a delivered order afterward.
- **New relations** added to support the reporting queries without denormalizing anything: `User::deals()` (`hasMany` via `sales_rep_id`), `Lead::deal()` (`hasOne`) тАФ both additive, neither changes any existing behavior.

### Bugs hit and fixed during this phase
1. **`Deal.value` was never populated from the quotation that actually justified it** тАФ see the Phase 3 amendment above. This was caught immediately by the very first tinker sanity check (`Deal::where('stage','won')->sum('value')` returned `0` despite a real Won deal with a 5,000 quotation existing), before any browser testing тАФ a good example of why this project checks raw data with `tinker` before trusting what a UI screenshot appears to show.
2. **`Company::withSum(['leads.deal as revenue' => ...], 'value')` тАФ Eloquent's nested "dot" relation aggregate does not support a filtering closure combined with an alias in the same call**; it threw `BadMethodCallException: Call to undefined method App\Models\Company::leads.deal()`, meaning Eloquent tried to resolve the whole dotted string as one relation method name instead of traversing it. Confirmed by testing the exact query in `tinker` before wiring it into the widget. Fixed by dropping the ORM aggregate helper entirely for this one query and writing a plain `join`-based query instead (`companies` тЖТ `contacts` тЖТ `leads` тЖТ `deals`, grouped and summed in SQL) тАФ more explicit, and sidesteps whatever internal limitation causes the alias+closure+dot-relation combination to fail.
3. Not a code bug, but worth recording: **`Filament\Widgets\Table`'s `->query()` only accepts a real Eloquent `Builder`**, so an idea to hand-aggregate in PHP and pass a plain array/collection to the table widget (which would have been simpler code) does not work with this API тАФ every table widget in this phase ended up as a single real query instead, which is also just more efficient than pulling every row into PHP to group there.

Verification: tinker sanity-checked every new query in isolation before it went into a widget (a `Deal::where('stage','won')->sum('value')`-style check for each). Then a full two-role-plus Playwright pass confirmed: every admin widget renders with its expected heading and the correct seeded numbers (including the top-clients table actually showing "Acme Packaging Co"), the agent dashboard shows the pipeline/conversion widgets and *not* any admin-only widget, and Production's dashboard is unchanged. Investigated an initial run that showed 9 stray `pageerror: Object` entries тАФ traced this to the test script leaving several already-visited Playwright pages open (each with its own live Echo/WebSocket connection and `wire:poll` timer from Phase 5) and only calling `browser.close()` once at the end, which aborts all of them simultaneously; confirmed harmless by explicitly closing each page first and re-running with zero errors, and separately confirmed the same artifact reproduces on an unrelated Phase 2 page (`/admin/leads`) with no Phase 6 code involved at all тАФ conclusively a test-teardown artifact, not an application bug, same root cause as the one already documented in Phase 5. Finished with a full regression run (`full-smoke-test.js`, covering all 5 roles and every nav page including Phase 5's chat/messages routes) тАФ zero HTTP 500s, zero unexpected console errors, only the two *expected* 403s from the access-control check.

## Example scenario: a lead's full journey to an order

This walks the exact path the system supports end-to-end, referencing what phase built each step тАФ useful both as a smoke-test script and as onboarding for anyone new to the app.

1. **Admin** logs in at `/admin/login`, goes to **Catalog тЖТ Products**, adds "5-ply Corrugated Box" (base price 50/unit) with a price tier: 500+ units at 40/unit. *(Phase 3)*
2. **Admin** goes to **System тЖТ Agents**, creates an agent "Fiona Full" with work-scope **Full Cycle**. *(Phase 2)*
3. **Fiona** logs in, goes to **Sales Pipeline тЖТ Leads**, clicks **New** on the "New" column, creates a lead for a new contact (inline company/contact creation right in the form) with product interest "Corrugated Boxes", assigns it to herself. *(Phase 2)*
4. Fiona works the lead: logs a call, adds a note, sets a follow-up date тАФ all from the lead's activity timeline. The Dashboard's **Follow-ups due** widget will surface it once that date arrives. *(Phase 2)*
5. Fiona drags the card to **Qualified**, then to **Quoted**. The moment it hits "Quoted", a **Deal** is silently created behind the scenes, owned by Fiona. *(Phase 3)*
6. Fiona goes to **Sales Pipeline тЖТ Deals**, finds the new deal card, opens it, clicks **New quotation**, adds the "5-ply Corrugated Box" line item, types quantity 600 тАФ unit price auto-fills to 40 (the 500+ tier) the moment she tabs out. Total and discount recalculate automatically. *(Phase 3)*
7. She clicks **Mark as Sent**. If the discount she'd negotiated exceeded the Admin-set threshold, this would instead tell her to wait for Admin approval тАФ it doesn't here, since there's no discount at list price. *(Phase 3)*
8. The client comes back positive. Fiona (or Admin) uses **Record client response тЖТ Approved by client**, then drags the Deal card to **Won**. This creates an **Order** (status Pending) and marks the Lead "Won" too. *(Phase 3)*
9. **Pat (Production)** logs in тАФ sees only **Production тЖТ Orders**, nothing else. Opens the order, clicks **Start production**, then **Mark ready to dispatch**, then **Dispatch** (fills in vehicle info, dispatch date, delivery address, invoice number), then **Mark delivered**. Fiona gets a bell notification at every step. *(Phase 4)*
10. Fiona opens **Chat**, clicks **Pat Production** in Direct Messages, types "Hey Pat, any update on the Acme order?" and sends. Pat тАФ logged in separately, already sitting on the Chat page with Fiona's DM open тАФ sees the message appear within a couple of seconds, no page reload, no click needed. Pat replies "Dispatched this morning, tracking number's on the order." Both sides now show the full back-and-forth, and it's saved in the `messages` table for Admin to audit later at **System тЖТ Chat Logs**. *(Phase 5)*
11. **Admin** opens the **Dashboard** and sees this whole deal reflected in the numbers: **Total revenue (Won deals)** now includes the 5,000 from Fiona's Acme quotation, **Revenue by agent** shows a bar for Fiona, **Top clients by revenue** lists Acme Packaging Co, and **Avg. order turnaround** reflects the WonтЖТDelivered time Pat's dispatch just produced. Fiona, on her own dashboard, sees **My open pipeline value** (0, since this deal is now Won and no longer "open") and **My conversion rate** (100% тАФ 1 won, 0 lost). *(Phase 6)*
